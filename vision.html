Chapter 01

The Problem

Why I chose automation over the status quo

In every role I've held, from telecom operations in Pakistan to semiconductor engineering in Austria, I've seen the same pattern: talented engineers spending most of their time on repetitive, manual tasks. Copy-paste configurations. Manual deployments. No monitoring. No version control. Status quo maintained because "that's how we've always done it."

I decided early in my career that I wanted to be the person who breaks that cycle — not by criticizing the process, but by building something better.

🔄

Manual Repetition

Engineers doing the same tasks by hand every day. No scripts, no automation, no templates. Hours lost to work that a well-written Python script could handle in seconds.

🔒

Siloed Knowledge

Critical know-how lived in people's heads, not in code or documentation. When someone left, the knowledge walked out with them. No runbooks, no shared understanding.

🐢

Slow Release Cycles

No CI/CD pipelines. Manual testing. Code shared via email or USB. Releases were rare, risky, and stressful events instead of routine, confident deployments.

🔇

No Observability

Systems running without proper monitoring. Problems discovered when users complained, not when metrics spiked. Reactive firefighting instead of proactive prevention.

📊

Fragmented Data

Data scattered across spreadsheets, file shares, and email threads. No structured pipelines. No data governance. Making decisions based on gut feeling instead of evidence.

🏗️

No Architecture Culture

Code written without documentation, without tests, without reviews. Technical debt accumulated silently until refactoring became impossible.

The Root Cause

The problems weren't technical — they were cultural. Organizations had the talent but lacked the automation mindset. My goal became simple: demonstrate that automation is not a threat to engineers, but a force multiplier that lets them focus on what matters.

Chapter 01

The Problem We Inherited

Not the process on a wiki what a Product Owner and a developer actually live through, every single cycle.
It is Monday morning. The Product Owner has a meeting with stakeholders, collects requirements, translates them into product requirements, manually writes every Jira ticket, assigns priorities based on tool usage data they carry in their head, and estimates effort with no tracking system to validate those estimates. Then they find the developer.

The developer opens Jira, picks up a ticket they barely understand no architecture documentation exists, no prior context is written down. They enter the development environment, write code in SKILL or Perl, implement the change, then manually test it because no test automation exists. They document the change themselves in the code or they don’t, because there is no standard. They push to a branch, open a pull request, wait for CI to run, fix whatever Jenkins or SonarQube flags, wait for review, fix again, wait again. Eventually it merges.

Twice a year, the Product Owner runs manual commands to cut a release. They update the viewspec. They open every Jira ticket individually, write a comment, close it. They run integration and end-to-end tests by hand. If something fails it goes back to the developer. If it passes they write the test documentation, finalize user documentation, and sometimes run a user training session. Then planning starts over from zero.

And then the phones start ringing. After every release, designers using the tools have questions. How does this function work? Why did the output look like this? What input format does this subflow expect? The Product Owner becomes the support desk answering the same questions from memory, every cycle. This is not a side task. It routinely consumes more time than the development work itself. This was 2022. Honestly, most of it is still 2026.
⚠️

Legacy Languages

SKILL and Perl cannot integrate AI tooling. The developer pool shrinks every year. New hires take months to become productive.

🚫

No Test Automation

Every change is validated by hand. The single biggest bottleneck to shipping faster.

📭

No Architecture Docs

Developers face a codebase with no map. Knowledge walks out the door when people leave.

🐌

Two Releases Per Year

Manual commands, viewspec updates, Jira cleanup all by hand. Release itself becomes a project.

📝

Manual Ticket Management

After every release, every Jira ticket is opened, commented, and closed individually. Manually.

🏝️

Siloed Subflows

No data flows between tools. No learning loop. No cross-subflow intelligence. No compounding knowledge.

🎧

Post-Release Support Overload

After every release the Product Owner becomes the support desk. Designers ask how tools work, what inputs are expected, why outputs look a certain way. The PO answers from memory the same questions, every cycle. This support burden often exceeds the time spent on development itself and has no scalable solution today.

Beyond our own domains, we play a crucial role integrating AI into legacy EDA systems, enabling EDA vendor AI (tools built on top of Cadence, Synopsys, Siemens) into our workflows, and contributing to cross-functional AI boards across Infineon.

The BDG Breakthrough Proof It Works

BDG functions lived inside Virtuoso SKILL code technically readable and editable, but built by experienced domain engineers whose deep knowledge was never documented. Today's developers face code shaped by decades of historical decisions with no written rationale: you could open the files, but understanding why things worked that way was another matter entirely. Batch-mode execution wasn't supported, so every bonding diagram required a human operating a GUI step by step. The conventional answer was: "You can't automate this."

1
AutoBDG

Python orchestration + XML input + DFIIBridge to call Virtuoso. Full BDG flow automated. Deployed for Promos.

2
BondwireGen

Automated module draws bondwires from user-defined pin inputs. Hours of manual drawing eliminated.

3
ProDiGI

AutoBDG v2 richer automation running hand-in-hand with BDG Assistant. Extracts design data into the pipeline, validates it for AI consumption, and guides the DDF team through order file creation using that data directly.

4
BDG AI Assistant

L1: Chatbot · L2: Craft order files · L3: Optimize designs + trigger DRC automatically.

The Repeatable Pattern

Take an encrypted, GUI-dependent, manually operated design subflow wrap it in Python connect it to Virtuoso through DFIIBridge feed it structured input then layer AI on top. This pattern shipped, runs in production, and applies to every subflow.

The Flywheel How It Accelerates

Each turn of the wheel makes the next turn faster. Software produces data. Data trains AI. AI improves software. The loop is self-reinforcing and it has already started.

BDG is already generating the first real data. ProDiGI is extracting and validating that data feeding it into the pipeline and helping the DDF team craft better order files. The flywheel has started turning.
